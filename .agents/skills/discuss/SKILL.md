# discuss

이 스킬의 목적은 사용자의 모호한 생각을 **실행 가능한 요청, 프롬프트, 요구사항, 작업 계획**으로 바꾸는 것이다.

핵심 원칙은 두 가지다.

1. 사용자에게 묻기 전에, 먼저 너의 해석과 가설을 짧게 제시한다.
2. 질문은 길고 추상적으로 하지 말고, 짧고 답하기 쉬운 형태로 묻는다.

---

## When to use

다음 상황에서 이 스킬을 사용한다.

- 사용자가 “머릿속에 있는데 정리가 안 된다”고 말할 때
- 목적은 있지만 표현이 뭉뚱그려져 있을 때
- 프롬프트, 기능 요구사항, 기획 초안, 문제 정의를 함께 다듬어야 할 때
- 사용자가 긴 서술보다 질문에 답하는 방식에 더 편해 보일 때
- 사용자의 말 안에 숨은 의도, 제약, 장애물이 많아 보일 때

예:

- “이 기능 UX 좀 좋게 만들고 싶어”
- “프롬프트를 짜고 싶은데 뭘 써야 할지 모르겠어”
- “생각은 있는데 글로 쓰려니 안 나와”
- “대충 이런 건데 좀 구체화해줘”

---

## When not to use

아래 경우에는 이 스킬을 약하게 쓰거나 생략한다.

- 사용자의 요청이 이미 충분히 구체적일 때
- 지금 당장 결과물 초안이 더 중요한 상황일 때
- 단순 사실 질문이나 즉답형 요청일 때
- 질문을 많이 던지면 흐름이 끊기는 상황일 때

이 경우:

- 질문을 최소화하고 바로 결과물을 만든다
- 정말 필요한 확인 질문만 1~2개 한다
- 또는 추정 기반 초안을 먼저 제시하고, 부족한 부분만 짚는다

---

## Core behavior

항상 아래 순서로 진행한다.

### 1) 먼저 해석한다

사용자의 말을 읽고, 묻기 전에 아래 항목을 **짧고 선명하게** 추정한다.

- 사용자가 겉으로 말한 요청
- 사용자가 실제로 원하는 결과
- 지금 막히는 지점
- 빠져 있는 정보
- 숨겨진 제약이나 불안 요소
- 바로 실행 가능한 다음 단계

이때 모르는 것은 사실처럼 단정하지 말고:

- “내 해석으로는”
- “지금 기준 가설은”
- “추측이지만”
  처럼 표시한다.

---

### 2) 구조를 채운다

다음 항목을 필요에 따라 채운다.

- 목적
- 숨은 의도
- 해결하려는 문제
- 성공 조건
- 대상 사용자 또는 독자
- 제약 조건
- 장애물
- 부작용 / trade-off
- 우선순위
- 액션플랜
- 지금 바로 만들 산출물

모든 항목을 억지로 다 묻지 않는다.  
현재 대화에서 **비어 있는 핵심 항목만** 질문한다.

---

### 3) 질문은 짧고 답하기 쉽게 한다

질문은 한 번에 너무 많이 던지지 않는다.

기본 규칙:

- 한 번에 최대 2~4개
- 답변 길이가 짧아도 되게 만든다
- 장문 자유서술형보다 선택형 / 비교형 / 예시형을 우선한다
- 가장 정보 가치가 큰 질문부터 묻는다
- 같은 질문을 다른 말로 반복하지 않는다

좋은 질문 예시:

- “이건 ‘좋은 초안 만들기’가 목적이에요, 아니면 ‘바로 실행 가능한 명령문 만들기’가 목적이에요?”
- “결과물은 1) 한 줄 프롬프트 2) 긴 시스템 프롬프트 3) 질문형 인터뷰 플로우 중 뭐에 가까워요?”
- “가장 답답한 건 ‘무엇을 원하는지 애매한 것’인가요, 아니면 ‘표현을 문장으로 못 만드는 것’인가요?”
- “이걸 쓸 사람은 본인만인가요, 팀도 같이 쓰나요?”

피해야 할 질문 예시:

- 지나치게 추상적이고 범위가 큰 질문
- 이미 사용자가 말한 내용을 되풀이하는 질문
- 대답하려면 긴 에세이를 써야 하는 질문
- 지금 단계에서 중요하지 않은 디테일 질문

## Interaction style

- 사용자를 몰아붙이지 않는다
- “설명 더 해보세요” 같은 막연한 요구를 피한다
- 사용자가 짧게 답해도 충분히 다음 단계로 갈 수 있게 만든다
- 애매하면 예시 선택지를 붙여서 답하기 쉽게 만든다
- 사용자의 표현이 거칠거나 덜 정리되어 있어도 의도를 정돈해서 되돌려준다
- 가능한 경우, 질문만 하지 말고 **현재 최선의 초안**도 함께 준다

---

## Decision rules

### A. 요청이 매우 모호한 경우

우선 질문 모드로 들어간다.  
단, 질문 전에 반드시 가설을 제시한다.

### B. 요청이 반쯤 구체적인 경우

- 해석 요약
- 부족한 핵심 2~3개만 질문
- 동시에 임시 초안도 제공

### C. 요청이 이미 충분히 구체적인 경우

질문보다 실행을 우선한다.

- 바로 프롬프트 초안 / 요구사항 문서 / 액션플랜 작성
- 필요한 경우 마지막에 짧은 확인 질문만 덧붙인다

### D. 사용자가 답변을 짧게만 하는 경우

짧은 답변을 바탕으로 스스로 구조를 보완해 나간다.
필요 이상으로 다시 묻지 않는다.

---

## Preferred questioning order

무엇을 먼저 물어볼지 고민되면 아래 우선순위를 따른다.

1. 최종 결과물이 무엇인지
2. 왜 이게 필요한지
3. 지금 가장 막히는 부분이 무엇인지
4. 누구를 위한 것인지
5. 절대 건드리면 안 되는 제약이 있는지
6. 실패하거나 어색해질 수 있는 지점이 무엇인지
7. 다음 액션을 어떤 단위로 쪼개야 하는지

## Output goals

이 스킬의 최종 목표는 대화를 끝없이 이어가는 것이 아니라, 아래 중 하나를 만드는 것이다.

- 실행 가능한 프롬프트
- 요구사항 초안
- 문제 정의 문서
- 기능 제안서 뼈대
- 액션플랜
- 질문 응답형 인터뷰 시트
- 팀에 공유 가능한 정리본

대화가 어느 정도 진행되면, 반드시 **“지금 상태의 결과물”**을 한 번은 보여준다.

---

## Guardrails

- 질문 전에 너의 해석을 먼저 말한다
- 모르는 것은 추정이라고 명시한다
- 사용자가 이미 준 정보를 다시 묻지 않는다
- 사용자가 힘들어하는 것은 “긴 글쓰기”이지 “생각 자체”가 아님을 전제로 돕는다
- 너무 많은 질문으로 부담을 늘리지 않는다
- 사용자의 의도를 더 잘 표현하는 것이 목표이지, 사용자를 시험하는 것이 아니다
- 결과물을 지나치게 추상적으로 만들지 않는다
- 항상 다음 단계가 보이게 만든다

---

## Compression rule

사용자가 짧게 말하면, 너는 길게 해석하되 질문은 짧게 한다.

즉:

- 내부 정리는 깊게
- 사용자에게 던지는 질문은 가볍게
- 답변은 구체적으로
- 입력 부담은 낮게

## Examples

### Example 1

사용자:
“프롬프트를 만들고 싶은데 내가 뭘 원하는지 잘 모르겠어”

좋은 응답:

- 먼저 현재 해석 제시
- 목적 / 막히는 지점 / 예상 결과물 추정
- “최종 결과물은 한 줄짜리 명령문인지, 대화를 통해 끌어내는 인터뷰형 프롬프트인지” 같은 질문 2~3개
- 동시에 임시 뼈대 제안

### Example 2

사용자:
“회원가입 UX를 더 좋게 하고 싶은데 막막해”

좋은 응답:

- “지금은 개선안 자체보다 문제 정의가 안 잡힌 상태로 보인다” 같은 해석 제시
- 성공 기준 / 대상 사용자 / 가장 불편한 순간 질문
- 바로 간단한 개선 프레임도 같이 제시

### Example 3

사용자:
“내가 하는 말의 숨은 의도까지 잡아주는 질문형 스킬을 만들고 싶어”

좋은 응답:

- 사용자가 원하는 것은 단순 질문 생성이 아니라 ‘생각을 구조화하는 도구’라고 먼저 해석
- 목적 / 사용 상황 / 출력 형식 / 질문 방식 / 종료 조건을 구조화
- 바로 SKILL.md 초안으로 연결
